Kinetic — AI-Powered Workout Coach
What to Build
Build a fully functional, mobile-first web application called Kinetic — an AI-powered workout coach that generates personalized weekly workout plans, tells the user exactly what to do each day, logs their performance, and adapts future workouts based on logged data. The core value proposition is a zero-decision daily flow: the user opens the app and is told exactly what to do. Every feature must preserve this.
Use React with Tailwind CSS, Supabase for auth + database, and the Anthropic Claude API for AI plan generation. Deploy as a modern web application.

App Architecture
Tech Stack

Frontend: React + Tailwind CSS + React Router
Backend/Database: Supabase (Auth, PostgreSQL, Row Level Security)
AI: Anthropic Claude API (called server-side via Supabase Edge Functions or an API route)
State Management: React Context + useReducer for workout session state; React Query or SWR for server data
Local Storage: Persist in-progress workout logging to localStorage for offline resilience

Database Schema (Supabase/PostgreSQL)
sql-- Users (handled by Supabase Auth, extend with profile)
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  onboarding_completed BOOLEAN DEFAULT FALSE,
  streak_count INTEGER DEFAULT 0,
  longest_streak INTEGER DEFAULT 0
);

CREATE TABLE onboarding_profiles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  goal TEXT NOT NULL CHECK (goal IN ('strength', 'muscle_gain', 'weight_loss', 'general_fitness')),
  experience_level TEXT NOT NULL CHECK (experience_level IN ('beginner', 'some_experience', 'intermediate')),
  equipment TEXT NOT NULL CHECK (equipment IN ('full_gym', 'dumbbells_only', 'barbells_and_dumbbells', 'bodyweight_only')),
  time_per_workout INTEGER NOT NULL CHECK (time_per_workout IN (30, 45, 60, 75)),
  days_per_week INTEGER NOT NULL CHECK (days_per_week IN (2, 3, 4, 5)),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE exercises (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  category TEXT NOT NULL, -- 'compound', 'accessory', 'isolation'
  muscle_group TEXT NOT NULL, -- 'chest', 'back', 'shoulders', 'legs', 'arms', 'core', 'full_body'
  equipment_required TEXT NOT NULL, -- matches equipment options
  description TEXT,
  movement_description TEXT -- 2-3 sentence how-to
);

CREATE TABLE weekly_plans (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  week_start_date DATE NOT NULL,
  status TEXT DEFAULT 'active' CHECK (status IN ('active', 'archived')),
  plan_data JSONB NOT NULL, -- full structured plan from AI
  generated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE workout_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  plan_id UUID REFERENCES weekly_plans(id),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  scheduled_date DATE NOT NULL,
  workout_title TEXT NOT NULL,
  status TEXT DEFAULT 'scheduled' CHECK (status IN ('scheduled', 'completed', 'missed')),
  started_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ,
  duration_seconds INTEGER,
  is_compressed BOOLEAN DEFAULT FALSE,
  is_retroactive BOOLEAN DEFAULT FALSE
);

CREATE TABLE prescribed_sets (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID REFERENCES workout_sessions(id) ON DELETE CASCADE,
  exercise_id UUID REFERENCES exercises(id),
  exercise_name TEXT NOT NULL,
  exercise_category TEXT, -- 'primary', 'accessory', 'isolation'
  set_number INTEGER NOT NULL,
  target_reps INTEGER NOT NULL,
  suggested_weight NUMERIC(6,2),
  sort_order INTEGER NOT NULL
);

CREATE TABLE logged_sets (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  prescribed_set_id UUID REFERENCES prescribed_sets(id) ON DELETE CASCADE,
  actual_reps INTEGER,
  actual_weight NUMERIC(6,2),
  skipped BOOLEAN DEFAULT FALSE,
  skip_reason TEXT CHECK (skip_reason IN ('equipment_unavailable', 'injury', 'time', 'other')),
  notes TEXT,
  logged_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE progression_events (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  exercise_name TEXT NOT NULL,
  event_type TEXT NOT NULL CHECK (event_type IN ('weight_increase', 'weight_decrease', 'volume_increase', 'deload')),
  old_value NUMERIC(6,2),
  new_value NUMERIC(6,2),
  reason TEXT,
  triggered_at TIMESTAMPTZ DEFAULT NOW()
);
Enable Row Level Security on all tables. Users can only read/write their own data.

3. AI Plan Generation (Claude API)
When the user completes onboarding (or a new week starts), call the Claude API to generate a structured weekly plan.
Prompt Strategy: Send the user's profile (goal, experience, equipment, days, time) plus any progression data to Claude with a system prompt that instructs it to return a strict JSON structure. Use a template-first approach: maintain a library of base plan templates and have Claude customize them.
Claude System Prompt (for plan generation):
You are a certified fitness coach AI. Generate a weekly workout plan as a JSON object.

User Profile:
- Goal: {goal}
- Experience: {experience_level}
- Equipment: {equipment}
- Time per workout: {time_per_workout} minutes
- Days per week: {days_per_week}
- Progression data: {progression_context}

Rules:
1. Return ONLY valid JSON, no markdown, no explanation.
2. Plan must cover exactly {days_per_week} training days and fill remaining days as rest.
3. Distribute rest days so no more than 2 consecutive training days before a rest day (except 2-day plans).
4. Do NOT front-load all training days.
5. Exercise selection must only use equipment the user has.
6. Volume per exercise:
   - Beginner: 2-3 sets, 8-12 reps
   - Some experience: 3 sets, 6-12 reps
   - Intermediate: 3-4 sets, 5-12 reps
7. Start weights conservatively for beginners.
8. Each workout should have 4-7 exercises depending on time available.
9. Include a mix of compound and isolation exercises appropriate to the goal.
10. Workout titles should be descriptive (e.g., "Upper Body Push", "Lower Body", "Pull Day", "Full Body").

Return this exact JSON structure:
{
  "days": [
    {
      "day": "Monday",
      "type": "training" | "rest",
      "title": "Upper Body Push",
      "estimated_duration": 45,
      "intensity": "moderate" | "high" | "low",
      "exercises": [
        {
          "name": "Bench Press",
          "category": "primary",
          "muscle_group": "chest",
          "sets": 3,
          "reps": 8,
          "suggested_weight_kg": 60,
          "rest_seconds": 90,
          "description": "Lie on bench, grip bar shoulder-width, lower to chest, press up."
        }
      ]
    }
  ]
}
Fallback: If the AI call fails or takes >8 seconds, serve a pre-built template plan matching the user's goal/experience/equipment combination. Show a note: "Your plan was created from a proven template — it will personalize as you log sessions."

8. Adaptive Progression Engine (Backend Logic — NOT AI)
This is deterministic rule-based code. Run after each completed session:
javascript// Progression Rules (implement as backend functions):

// Rule 1: Weight Increase
// If user completed ALL prescribed reps for ALL sets of an exercise 
// at the same weight for 2 CONSECUTIVE sessions → increase weight.
// Compound lifts: +2.5 kg, Isolation: +1.25 kg

// Rule 2: Weight Decrease  
// If user failed to complete minimum target reps for 2 CONSECUTIVE 
// sessions → reduce weight by one increment.

// Rule 3: Volume Increase
// After 3 consecutive weeks of completing ≥75% of scheduled sessions 
// → add 1 set to primary exercises (max ceiling applies).

// Rule 4: Deload
// If 2+ sessions missed in a week → next week volume reduced ~15-20%.

// Rule 5: Proactive Increase
// If user consistently logs significantly more reps than prescribed 
// (e.g., prescribed 8, logs 12+) for 2 sessions → increase weight.

// Edge Cases:
// - Partial workouts: only apply progression to exercises with complete data.
// - <50% sessions logged in 2 weeks: pause progression, hold current weights.
// - First 2 sessions: no progression applied (baseline period).
// - 7+ day gap: treat next session as fresh baseline.
Store all progression events in the progression_events table for transparency.

Edge Cases & Error Handling

Plan generation failure: Show error after 10s timeout, "Try Again" button. After 3 failures, show template fallback.
Network loss during workout logging: All logging state persists to localStorage in real-time. Banner notification "You're offline — data saved locally." Auto-sync on reconnection.
Rest day landing: Show rest day card with moon icon, motivational message, and upcoming sessions.
No plan exists: Redirect to onboarding.
Duplicate session prevention: If today's session is already completed, show "Completed ✓" state. Option to log supplemental session.
Returning after 14+ days: Show "Welcome back!" prompt with options: "Fresh Start" (re-onboarding) or "Continue" (resume with deload).
Retroactive logging: Allow logging yesterday's missed session only (marked as "Logged late").
